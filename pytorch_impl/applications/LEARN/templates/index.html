Garfield is a research project developed at Prof. Rachid Guerraoui's Distributed Computing Lab, at EPFL. It is a framework to write Machine Learning applications, built on top of PyTorch and TensorFlow, two of the most widely used platforms in this field.

Garfield focuses on Distributed ML, which is a way to use multiple machines that work together on training a model with data. This can be done for various reasons, such as efficiency or the preservation of privacy, and leads to diverse distributed architectures and computing algorithms. Garfield can work with any of these architectures, and provides examples for many of the common ones:

When working with several machines concurrently, the risk of failures increases. These can negatively impact the computed results if the system is not ready to handle them. Besides providing an abstraction to develop distributed ML applications, Garfield seamlessly offers means to deal with generic failures (known as Byzantine Faults). Using Garfield, developers can easily build applications that are tolerant to such failures and still give meaningful results.

This demonstrator will focus on the decentralized architecture. It will guide you through training a model, examining how learning evolves when we add more machines, what impact failures can have, and how Garfield helps to manage them.

[[ stepTitle ]]

The data used throughout the demonstrator comes from the Pima Indian Diabetes dataset. It consists of several medical values from female patients (such as number of pregnancies, blood pressure, age, BMI, etc.), along with an indication of whether they had been diagnosed with diabetes.

To determine how good a model is, we proceed as follows. First, we train it over a subset of the available data. After that, we use another subset to measure the accuracy. More precisely, we use the model, computed over the first subset, to predict the diabetes diagnostic on data from the second subset. By comparing these predictions with the actual known information, we can estimate the model's accuracy.

In this first step, we will train a model using a single machine.

This will give us a reference value, to serve as a basis for comparison in the next steps.

By using more data to train a model, we are likely to obtain a more accurate model. Therefore, ideally, we gather all the available data in a single large dataset, and train our model over it. We can then use a distributed architecture with many machines running in parallel to perform the computations faster.

In some situations, however, it is not possible nor desirable to combine all the data centrally. For example, patient medical data from different hospitals cannot generally be combined, due to the requirements to preserve patient privacy. Similarly, different financial institutions will not share their data for business or competitive reasons.

In these cases, it is still possible to benefit from the larger dataset by using a decentralised architecture known as Federated learning. With this approach, each machine trains a model with their locally available data, and the model updates (the gradients) are then exchanged between machines, combined, and used to feedback into the next learning loop. This allows each system to reach a model over the global data without exchanging the data itself.

We will now add more machines to our computation, forming a distributed system. Each machine has its specific subset of the data. During the training, the model updatess are exchanged between the systems, improving the local model. In the end, the accuracy is measured over the same test subset as before.

So far, we have assumed that all of our systems are reliable, and always behave correctly. Unfortunately, this is not a realistic expectation. In the real world, systems can fail for a variety of reasons: hardware malfunctions, programming defects, human errors, data corruption, deliberate attacks, etc. To capture the generic concept of errors stemming for any possible cause, inducing a behavior that deviates arbitrarily from the correct one, we use the term of Byzantine faults.

If failures happen during a distributed learning computation, and our system is not programmed to handle them, the resulting model is likely to become significantly less accurate.

In this step, to simulate failures, we will have some of the systems return random data in place of the model updates when exchanging data during training. We will not change the computation, which "naively" assumes all systems are reliable.

As briefly mentioned in the introduction, Garfield is designed to develop distributed ML algorithms, and provides means to handle system failures, all the while preserving the model accuracy as much as possible. In other words, Garfield makes our algorithms transparently Byzantine-resilient.

The way Garfield achieves this is by using robust rules to combine results from different sources, known as aggregators, which are able to filter out the Byzantine effects. Which aggregator to use, and what kinds of effects they can tolerate, depends on the system parameters and data in use. We will not elaborate on the details here, but interested parties will find this discussed at length in the project paper.

We will now make use of Garfield's ability to train a model accurately even though some of the systems exhibit failures.

Please note that the data and robust aggregator used for this demonstrator is only able to tolerate less than half of the machines being unreliable, so depending on your parameter choices you will observe varying degrees of success. Feel free to go back a few steps, change the parameters and rerun the trainings if you wish to eperiment different scenarios.

Please also note that, due to the different computation used in this step, along with the random nature of model training, it is possible that the accuracy achieved in this step slightly exceeds the one obtained with all reliable systems. It should nonetheless be within the same ballpark.

This concludes the demonstrator, which hopefully gave you a feel of what Garfield can achieve and how it can help you develop more robust distributed ML applications.

For training a model over several datasets in a privacy-preserving context, other approaches are possible. Depending on your use-case and your requirements, it might be worth to also take a look at a solution such as SPINDLE, which employs different techniques and makes other trade-offs.

For example, with Garfield, even though only model updates are shared between systems, sophisticated attacks exist that can leak information about the data. In other words, an attacker might be able to retrieve information about a system's data solely by analyzing the exchanged model updates.

By contrast, SPINDLE uses Secure Multiparty Computation and Homomorphic Encryption to provide more privacy-preserving guarantees. On the other hand, this comes with limitations: these techniques are much more CPU-intensive and require enormous amounts of storage, making them currently difficult to use with very large datasets.

Depending on your needs, you will find one or the other method more appropriate for your application. We will be happy to discuss with you in further details the pros and cons of each of these solutions, and help you in your decision. Don't hesitate to contact us!
